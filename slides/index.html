Concepts · 4 / 6

Similarity is just distance

Cosine distance

def cosine_dist(a, b):
    return 1 - dot(a, b) / (norm(a) * norm(b))

Close to 0 = very similar. Close to 1 = unrelated.

In Postgres / AlloyDB

-- cosine distance operator
SELECT description
FROM cymbal_products
ORDER BY embedding <-> query_vector
LIMIT 5;

The <-> operator is from the vector type.

Sorting by similarity = sorting by distance. That's the whole trick.

Concepts · 6 / 6

Why "vectors inside Postgres" is a big deal

  • One source of truth — no syncing product data between Postgres and a vector DB
  • Joins just work — filter by price < 1000 and semantic relevance in one SQL
  • Transactions — embedding updates are ACID with the rest of your data
  • No glue code — no Python service standing between your app and search
The alternative is a "sync nightmare": product updated in Postgres, vector stale in Pinecone. Been there — it's not fun.

SAY: "Ab specifically — Postgres ke andar vectors rakhne ka bada fayda kya hai?"

SAY: "Pehla — one source of truth. Product details aur vectors same database mein. Koi sync ka tension nahi."

SAY: "Dusra — joins seamlessly work karte hain. 'Price 1000 se kam' plus 'semantic match' — dono ek hi SQL mein."

SAY: "Teesra — transactions atomic hain. Pinecone use karoge toh Postgres update hua, Pinecone nahi — sync nightmare."

SAY: "Real project mein maine ye pain face kiya hai. Vector DB aur main DB ke beech sync — not fun at all."

Kubernetes Operators, one paragraph

An Operator is a robot DBA that lives inside your cluster.

  • You write a tiny YAML that says "I want a DB cluster like this".
  • The Operator watches that spec.
  • Spawns pods, configures backups, handles failover, keeps reality = spec.

The AlloyDB Omni operator is how we avoid writing 400 lines of K8s manifests.

The stack · 4 / 5

EmbeddingGemma, served by TEI

The model
  • EmbeddingGemma — Google's open embedding model
  • 308M parameters · 768-dim output
  • 2K token context · multilingual
  • Hugging Face hosted, Apache 2.0
The server
  • TEI — Text Embeddings Inference (by Hugging Face)
  • Runs the model as an HTTP service
  • One container image, one pod in GKE
  • AlloyDB Omni calls it via internal cluster DNS
Why local, not Vertex AI Embeddings API? No quota · no per-call bill · no data leaves your cluster · privacy by default.

EXPLAIN ANALYZE · the real story

Where the time actually goes

Real EXPLAIN ANALYZE from the hero query. Total wall time: 20,160 ms.

At 941 rows, a sequential scan was fine. At 10 M rows, add CREATE INDEX ... USING hnsw (embedding vector_cosine_ops) — same SQL, sub-10 ms stays.
Real production fix: cache or pre-compute query embeddings. 99.97% of the cost is embedding generation, not search.
